## Architecture summary

Four-layer model (ADR-025): Platform → Deployable Services → Runtime Concepts → Business Concepts.

```
Spatial Runtime Platform
├── Deployable Services: Gateway | Room Service | Game Server
└── Runtime Concepts: Space → Runtime Node → Zone → Entity → AOI → Ownership
```

**Service boundaries:** Gateway terminates WS + routes; Room Service owns runtime registry + node identity + zone allocation via `Allocator` interface; Game Server hosts Runtime Nodes executing Spaces, owns entity lifecycle + AOI + position sync.

**Key runtime contracts:** `Lifecycle` (per-entity extension), `Event`s (outbound observation), opaque `Attrs` + string `Type` (data extensibility). No plugin framework — interface-based extension.

## Production hardening completed

- **Scheduler:** bounded tick, measured wall-clock dt (capped 250ms), control-command drop protection.
- **Backpressure:** bounded queues (Inbox/Events/cmds), overflow drops counted + exported.
- **Observability:** all 10 Prometheus metrics wired (TickDuration, Overruns, QueueDepth×3, Dropped, EntityCount, ActiveSpaces, RuntimeEvents, PacketsPerSec, ActiveConnections, gRPC request duration).
- **Graceful shutdown:** gateway drain, relay context cancellation, dial timeout.
- **Node identity:** `NodeDescriptor` with immutable `NodeID` + `AdvertiseAddr`, `Allocator` interface (LeastLoaded implemented), richer heartbeat (load for scheduling).

## Known limitations

- **Multi-node scaling beyond 2 nodes:** runtime model supports it (NodeDescriptor, Allocator, peer registry, ghost store). Compose multi-node (docker-compose.multinode.yml) defines 2 nodes. Next phase: 4→8 nodes with ownership-transfer validation.
- **Race detector on benchmark harness:** event‑drainer goroutine shares entity/pattern slices with test assertion without synchronization (bench gear, not runtime core). No logical errors; fix is low‑priority mechanical.
- **Snapshot:** the per‑zone JSON snapshot does not include `RuntimeID`; the FK `zone_state_runtime_id_fkey` constraint causes a non‑fatal snapshot‑save WARN for demo‑seeded zones (no real runtime). Snapshot path is on‑disk (storage/game), not hot‑path.
- **pprof endpoints on services:** not exposed on container ports yet (service binaries only serve `/metrics`). CPU/heap profiling via `go test -bench` flags covers the in‑process path.

## Recommended next phase

1. **Chaos Engineering** — failure injection (node crash, gateway/room‑service restart, network partition).
2. **Distributed Scale Validation** — 4→8 Runtime Nodes, ownership transfer stress, cross‑node AOI at scale.
3. **Autoscaling** — hook the `Allocator.Select` and `NodeLoad` heartbeat into HPA‑style capacity decisions.
4. **Production Candidate** — add `/debug/pprof` to services, run long‑duration mem/goroutine leak tests.
