## 1. Architecture Overview

### Recommended Approach: Hybrid — Lightweight Coordinator + Direct P2P RPC

```
                    ┌─────────────┐
                    │   Clients   │
                    └──────┬──────┘
                           │ WebSocket (TLS + JWT)
                    ┌──────▼──────┐
                    │   Gateway   │  stateless, horizontally scalable
                    └──────┬──────┘
                           │ gRPC (lookup)
                    ┌──────▼──────┐
                    │Room Service │  HA coordinator (leader/follower)
                    │metadata only│  zone → GameServer mapping
                    └──────┬──────┘
                     ┌─────┼─────┐
                     │     │     │
               ┌─────▼┐ ┌─▼──┐ ┌▼─────┐
               │Game 1│ │Game│ │Game N│
               │zone A │ │zone│ │zone  │
               │zone B │ │ C  │ │ D..F │
               └──┬───┘ └─┬──┘ └──┬───┘
                  │        │       │
                  └───gRPC(direct)─┘
                    direct P2P RPCs
               ┌─────┴─────────┴──────┐
               │  PostgreSQL (source)  │
               │  Redis (cache)       │
               └──────────────────────┘
```

### Architecture Principles

1. **Production First** — Architecture designed for final production topology from day one. Implementation phases never change architecture.
2. **Horizontal Scaling First** — Prefer adding instances over increasing CPU/RAM.
3. **Logical Service Independence** — Service definition independent from physical deployment. No code changes needed when topology changes.
4. **Infrastructure as Code** — Everything reproducible from source. No manual production deployment.
5. **Cloud Agnostic** — No business logic depends on a specific cloud provider.
6. **Clean Separation** — Spatial Server = realtime infrastructure. Business Backend = business logic. Never mix.
7. **Single Source of Truth** — ADRs document decisions. Design doc is authoritative reference.
8. **Benchmark Driven** — Performance targets defined before optimization. No production without measurement.

### Platform Boundary

Spatial Server is **NOT** a business backend. It is a reusable realtime infrastructure platform.

**Business Backend owns:**
- Authentication (issues JWT tokens)
- User management, organizations
- Room/showroom/meeting metadata
- Access control, permissions
- REST API, admin dashboard

**Spatial Server owns:**
- Runtime instance lifecycle
- Zone ownership and Game Server allocation
- AOI and entity replication
- Player presence and movement
- Real-time synchronization

Spatial Server **never** contains business logic, user data, or room metadata.

### Component Responsibilities

| Service | Role | State |
|---------|------|-------|
| **Gateway** | WebSocket termination (nhooyr.io), client auth, rate limiting, connection routing | Stateless |
| **Room Service** | Zone ownership table, load balancing, service discovery, HA coordination | Lightweight metadata |
| **Game Server** | Entity simulation, AOI queries (in-memory), state persistence, client state replication | Zone state |
| **PostgreSQL** | Runtime metadata, zone ownership records, Game Server registry | Source of truth (operational only) |
| **Redis** | Session cache, metadata cache, pub/sub for non-realtime domain events | Cache layer |

### 3.4 Production Principles

- Infrastructure is immutable. Servers are never configured manually.
- Everything reproducible from source code (Terraform + cloud-init + Helm).
- SSH is for debugging only, never for deployment.
- Terraform provisions VMs, networking, DNS, volumes, LBs — never installs apps.
- cloud-init bootstraps new servers: Docker, K3s, cluster join.
- Helm charts exist for every deployable component (gateway, room-service, game-server, redis, postgres, monitoring).
- Secrets never in source code. Kubernetes Secret + env vars.
- Logs are structured JSON with trace_id, request_id, correlation_id.

## 5. Scaling Strategy

### Gateway — Horizontal, Stateless

```
Scale: add/remove instances behind load balancer
No session affinity needed (client routed by Room Service lookup)
Health: /health (TCP or HTTP)
```

### Room Service — Leader Election, HA

```
Active/Passive pair
Production: Kubernetes Lease API (coordination.k8s.io)
Dev: single instance (no election)
Leader handles: zone assignment, load balancing, service discovery
Follower: warm standby, accepts readiness probes
Failover: <5 seconds
```

### Game Server — Horizontal, Coordinator-Managed

```
Room Service decides when to add/remove Game Servers
New Game Server joins → Room Service assigns orphan zones
Game Server drains → Room Service reassigns zones to others
Game Server crash → Room Service detects via heartbeat timeout, reassigns zones
```

### Redis — Standalone → Sentinel → Cluster

```
Local dev:  standalone single instance
Staging:    Sentinel (1 primary, 2 replicas)
Production: Redis Cluster (data sharded across nodes)
```

### PostgreSQL — Primary + Replica

```
Local dev:  single instance
Staging:    primary + 1 replica (reads)
Production: primary + N replicas (read replicas for analytics)
Write path: always primary
Read path:  primary for cache misses, replicas for analytics
```

## 6. Autoscaling Strategy

### Metrics

| Metric | Threshold | Action |
|--------|-----------|--------|
| CPU (Game Server) | >70% for 30s | Spawn new Game Server |
| Memory (Game Server) | >80% for 30s | Spawn new Game Server |
| Connections per Gateway | >10K for 30s | Spawn new Gateway |
| Room count per Game Server | >N (configurable) | Reassign zones |
| Zone ownership imbalance | stddev > threshold | Rebalance zones |

### Scaling Flow: New Game Server Joins

1. New Game Server starts, connects to Room Service
2. Room Service registers it, marks it as `JOINING`
3. Room Service identifies most loaded existing Game Servers
4. Room Service selects zones to transfer (least loaded zones first)
5. Room Service sends `TransferZone(zoneID, targetID)` to source Game Server
6. Source Game Server serializes zone state (entities, positions, AOI data)
7. Source Game Server sends zone snapshot to target Game Server (direct gRPC)
8. Target Game Server loads snapshot, starts simulation
9. Target Game Server confirms ownership → Room Service updates ownership table
10. Gateway refreshed with new routing table (pushed or polled)

### Scaling Flow: Zone Ownership Change

```
1. Room Service detects imbalance
2. SELECT zones FROM game_server WHERE load > threshold
3. Pick least loaded destination Game Server
4. Notify source: prepare zone for transfer (pause AOI updates)
5. Transfer zone state via direct gRPC (source → target)
6. Atomic ownership switch in Room Service (PostgreSQL transaction)
7. Notify Gateway of routing change
8. Source: release zone resources
```

## 7. Service Discovery

### Chosen: Room Service as Discovery Authority + DNS fallback

**Why not gossip / etcd / Consul:**

- Simplicity: no external dependency for cluster state
- Room Service already exists as the coordinator
- DNS provides static bootstrap entry point
- Suitable for initial deployment scale

### Discovery Flow

```
1. Game Server starts, resolves room-service via DNS (static or K8s Service)
2. Game Server registers with Room Service (type, address, metadata)
3. Room Service maintains in-memory registry + PostgreSQL backup
4. Gateway asks Room Service: "who owns zone X?" → "Game Server at 10.0.1.5:9001"
5. Gateway caches routing table (TTL: 5s, pushes on change)
6. Game Servers discover each other via Room Service (for direct P2P RPC)
```

|----------|--------|

### Internal (Gateway ↔ Game Server)

| Strategy | Detail |
|----------|--------|
| Direct (no LB) | Gateway gets exact Game Server address from Room Service |

## 9. RPC Design

### Gateway → Room Service

| RPC | Type | Timeout | Retry | Idempotent |
|-----|------|---------|-------|------------|
| `LookupZone(zoneID) → GameServerAddress` | Unary | 2s | 3x | Yes |
| `ReportMetrics(load) → OK` | Unary | 1s | No | No (best effort) |

### Room Service → Game Server (control plane)

| RPC | Type | Timeout | Retry | Idempotent |
|-----|------|---------|-------|------------|
| `TransferZone(zoneID, snapshot) → OK` | ServerStream | 30s | 1x | Yes |
| `PrepareShutdown() → OK` | Unary | 5s | No | Yes |
| `Heartbeat() → OK` | Unary | 2s | No | Yes |

### Game Server ↔ Game Server (data plane)

| RPC | Type | Timeout | Retry | Idempotent |
|-----|------|---------|-------|------------|
| `SendEntityUpdate(entityID, pos, attrs)` | Unary | 500ms | No (latest wins) | No |
| `QueryEntities(aoiBounds) → EntityList` | Unary | 1s | 1x | Yes |
| `MigrateEntity(entityID, snapshot)` | Unary | 5s | 2x | Yes |
| `ZoneStateSync(snapshot)` | ServerStream | 30s | 1x | Yes |

### Protocol Design

```
gRPC (HTTP/2)
├── Proto3 definitions in /proto/
├── Compression: gzip (configurable per RPC)
├── Timeout: per-RPC configurable (defaults above)
├── Retry: exponential backoff (100ms base, max 3s)
├── Streaming: ServerStream for zone transfer, entity sync streams
└── Auth: mTLS optional (internal network), JWT for room-service ↔ gateway
```

## 15. Security

| Layer | Mechanism | Detail |
|-------|-----------|--------|
| **Auth** | JWT (issued by Business Backend) | Client presents runtime token on WebSocket connect. Gateway validates signature using Business Backend's public key. |
| **Token Contents** | player_id, runtime_id, exp | Spatial Server never issues tokens. Never manages users. |
| **Transport (ext)** | TLS 1.3 | WebSocket over WSS. Certificate from Let's Encrypt / cert-manager. |
| **Transport (int)** | TLS (optional) | gRPC over mTLS optional within private network. Not required for MVP. |
| **Rate Limiting** | Token bucket per IP | Gateway enforces: 100 msg/s per connection, 500 msg/s per IP. Configurable. |
| **Replay Protection** | Nonce + timestamp | Each client packet includes monotonic sequence number. Gateway rejects duplicates within 30s window. |
| **Packet Validation** | Schema validation | All client messages validated against protobuf schema before processing. Reject malformed packets. |

## 16. Failure Recovery

| Failure Scenario | Detection | Recovery |
|-----------------|-----------|----------|
| **Gateway Crash** | Health check fails | LB routes to other Gateway. Clients reconnect via session recovery (resume with session token). |
| **Room Service Crash** | Health check fails | Follower becomes leader (Kubernetes Lease API). Ownership table from DB. Gateway cache still valid for up to 5s. |
| **Game Server Crash** | Heartbeat timeout (15s) | Room Service marks zones as orphan. Assigns to available Game Servers. New owner loads zone state from PostgreSQL. Ghost entities cleaned on next AOI sweep. |
| **Redis Crash** | Connection error | Graceful degradation: Game Server reads from PostgreSQL directly (slower but correct). Redis repopulates on restart. |
| **PostgreSQL Crash** | Connection error | All services operate in degraded mode (in-memory only). Reads from cache. Writes queued. On recovery, replay queued writes. |
| **Network Partition** | Heartbeat timeout on both sides | Split-brain prevention: Room Service marks zone as LOST, assigns new owner. Old owner detects partition ends, surrenders zone. PostgreSQL acts as tiebreaker (advisory lock). |

## 17. Ownership Design

### Zone Ownership

```
Zone: a grid cell within a runtime.
Size: configurable (default 100x100 units).
Owner: exactly ONE Game Server owns a zone at any time.
```

| Operation | Owner | Mechanism |
|-----------|-------|-----------|
| Claim zone | Room Service assigns | PostgreSQL transaction: `INSERT INTO zone_ownership (zone_id, server_id) VALUES (?, ?)` |
| Transfer zone | Room Service initiates | Two-phase: PREPARE (pause zone) → COMMIT (new owner starts) |
| Release zone | Room Service on drain | `DELETE FROM zone_ownership WHERE zone_id = ?` |
| Recover orphan | Room Service on timeout | `UPDATE zone_ownership SET server_id = ? WHERE heartbeat_expires < NOW()` |

### Ownership Transfer

```
1. Room Service decides to transfer zone Z from Server A to Server B
2. Room Service writes: zone_ownership.zone_id = Z, .status = 'TRANSFERRING'
3. Room Service notifies A: "transfer zone Z to B"
4. Server A serializes zone state → sends to Server B (direct gRPC)
5. Server B loads state, starts processing
6. Server B acknowledges → Room Service updates: .status = 'ACTIVE', .server_id = B
7. Room Service notifies Gateway of routing change
```

### Conflict Resolution

```
Two Game Servers claim the same zone:
→ PostgreSQL unique constraint on zone_id prevents duplicates
→ First INSERT wins, second gets error
→ Conflict resolved by database, no ambiguity
```

### Recovery After Partition

```
Partitioned server S held zone Z.
S's heartbeat expires in Room Service.
Room Service assigns zone Z to server T.
When S comes back online, S discovers zone Z is no longer owned by S.
S surrenders zone Z, cleans up local state.
```

## 18. Runtime Lifecycle

### Flow

1. **Business Backend** creates a room/showroom in its own database.
2. **Business Backend** calls Spatial Server: `CreateRuntime(runtime_id, zone_count)`.
3. **Room Service** allocates zones, assigns Game Servers, stores runtime metadata in PostgreSQL.
4. **Business Backend** receives `gateway_addr` and returns it to the client.
5. **Client** connects to Gateway with runtime token (JWT containing: player_id, runtime_id, exp).
6. **Gateway** validates token signature → looks up runtime zone → proxies to Game Server.
7. **Client** participates in realtime simulation.
8. When room ends, **Business Backend** calls `DestroyRuntime(runtime_id)`.
9. **Room Service** marks runtime as destroyed, releases zones, disconnects players.

### API (SpatialServerAPI)

| Method | Description |
|--------|-------------|
| `CreateRuntime` | Allocate zones and Game Servers for a new runtime |
| `DestroyRuntime` | Release all resources for a runtime |
| `GetRuntimeInfo` | Query runtime status, player count, zone count |
| `GetRuntimeMetrics` | Query runtime performance metrics |
| `ListRuntimes` | List all active runtimes |

### States

| State | Description |
|-------|-------------|
| `creating` | Zones being allocated, Game Servers being assigned |
| `active` | Runtime is running, players can connect |
| `draining` | Runtime is being destroyed, players being disconnected |
| `destroyed` | All resources released |

### Ownership

- Runtime is owned by the Business Backend that created it.
- Spatial Server does not enforce multi-tenant isolation (trusted Business Backend model).

## 19. AOI Design

| Parameter | Default | Rationale |
|-----------|---------|-----------|
| **Grid Size** | 100x100 units | Balances zone count vs. entity density |
| **Update Frequency** | 20Hz (matches tick rate) | Entities broadcast position every tick |
| **Interest Radius** | 300 units (3x3 grid) | Players see entities in adjacent zones |
| **Ghost Entity** | 500ms timeout | Entity crosses zone boundary → broadcast "enter" to new zone, schedule "leave" from old zone after 500ms |
| **Replication** | Subscribe to adjacent zones | Each Game Server subscribes to AOI updates from all zones within interest radius |
| **Storage** | In-memory (Game Server) | Realtime AOI state never passes through Redis. Redis only stores metadata. Zone transfer serializes in-memory AOI state. |

### AOI Query Flow

```
1. Entity E moves within zone Z
2. Game Server G updates E's position in zone Z's in-memory AOI index
3. If E crosses zone boundary:
   a. G broadcasts "entity_leave" to old zone's subscribers
   b. G sends "entity_enter" to new zone's owner
   c. New zone owner adds E to its AOI index
4. G calculates entities within E's interest radius (AOI query)
5. G sends position update to all interested clients (via Gateway)
```

```
Per-packet gzip compression (configurable)
Flag bit 0 indicates compressed payload
Compression level: 3 (balance speed vs. ratio)
```

### Backpressure Strategy

When Game Server or Gateway is overloaded:
1. Drop non-critical updates (position interpolation, cosmetic animations)
2. Reduce broadcast frequency (AOI updates skip every Nth tick)
3. Disconnect abusive clients (rate limit violators)
4. Never block the Game Loop — backpressure must be non-blocking

### Backward Compatibility

```
New packet IDs can be added without breaking old clients
Clients ignore unknown packet IDs
Server communicates supported protocol range during handshake
```

## 21. Coding Standards

### Naming

| Convention | Rule |
|------------|------|
| Packages | lowercase, single word (`entity`, `space`, `zone`) |
| Files | snake_case, matching package (`entity.go`, `entity_test.go`) |
| Exported types | PascalCase (`EntityID`, `ZoneManager`) |
| Unexported | camelCase (`zoneOwnership`) |
| Interfaces | PascalCase, -er suffix when applicable (`Storage`, `AOIIndex`) |
| Constants | PascalCase (`MaxPlayersPerZone`) |
| Tests | `TestXxx` (standard Go), `TestXxx_GivenY_WhenZ` (table-driven) |

### Package Layout

```
pkg/<domain>/
├── <domain>.go        # Main types + interface
├── <domain>_test.go   # Unit tests
├── impl/              # Implementations (if interface is in parent package)
│   └── <variant>.go
or
└── <variant>.go       # Single implementation file
```

### Interface Rules

```
- Define interfaces in the consumer package (not implementor)
- Prefer small interfaces (1-3 methods)
- Accept interfaces, return concrete types
```

### Dependency Rules

```
apps/* → pkg/* → internal/ (never the reverse)
pkg/* depends on: standard library, Google gRPC, slog, protobuf
pkg/* never depends on: apps/*, HTTP frameworks (except in adapter packages)
Infrastructure abstractions in pkg/storage (PostgreSQL/Redis adapters in subpackages)
```

### Error Handling

```
- Sentinel errors for expected failures (var ErrNotFound = errors.New("not found"))
- Wrap errors with context: fmt.Errorf("lookup zone %s: %w", zoneID, err)
- Log at the boundary (RPC handler, goroutine entry point), not in business logic
- Panics only for truly unrecoverable states
```

### Testing

```
- Unit tests alongside source (pkg/xxx/xxx_test.go)
- Integration tests in tests/integration/
- Mock interfaces via GoMock or hand-written
- Table-driven tests preferred
```

## Appendix B: Glossary

| Term | Definition |
|------|------------|
| **Zone** | A grid cell within a runtime. The atomic unit of ownership. |
| **Runtime** | An instantiated realtime session (corresponding to a business room). Composed of one or more zones. |
| **Entity** | A simulated object within a runtime (player avatar, NPC, interactive object). Has position, attributes, and a type. |
| **AOI** | Area of Interest. The set of entities within a player's visual range. |
| **Gateway** | Edge service that terminates WebSocket connections from clients. |
| **Room Service** | Coordinator service that manages zone ownership and load balancing. |
| **Game Server** | Core service that simulates entities within its owned zones. |
| **Ownership** | The binding of a zone to exactly one Game Server at any time. |
| **Zone Transfer** | The process of moving zone ownership from one Game Server to another. |
| **Ghost Entity** | An entity whose owner has moved to a new zone but is still visible in the old zone for a brief period (smooth transition). |
